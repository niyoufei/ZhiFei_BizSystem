## 4. 当前剩余未跟踪项分类

当前 main worktree 仍存在大量未跟踪项，需按类别治理。

### 4.1 `.playwright-cli/`

`.playwright-cli/` 属于浏览器 / 自动化工具相关缓存或产物。

处置原则：
- 需先确认来源、生成时间、是否仍需复现浏览器验证；
- 不直接删除；
- 如判定可清理，也应先记录规模和样例文件，再单独授权处理。

### 4.2 `data/`

`data/` 属于高风险目录，可能包含业务数据、样本数据、运行态数据或临时验证数据。

处置原则：
- 禁止直接删除；
- 禁止直接移动；
- 禁止提交真实业务数据入仓；
- 后续必须先只读盘点文件类型、数量、大小和敏感性；
- 如需保留，应优先仓库外备份。

### 4.3 `docs/final/` 与 `docs/next/`

`docs/final/`、`docs/next/` 可能是交付物、阶段文档、恢复说明或历史工作记录。

处置原则：
- 禁止直接删除；
- 禁止直接移动；
- 先盘点文件名、数量、修改时间和内容主题；
- 再判断是否属于正式交付、阶段草稿、历史恢复文档或可归档资料。

### 4.4 `output/`

`output/` 可能是导出产物、报告产物、运行态输出或调试结果。

处置原则：
- 禁止直接删除；
- 先确认是否包含业务报告、生成结果或可复核证据；
- 如需要清理，应先做仓库外备份或形成保留 / 删除清单。

### 4.5 `tmp/`

`tmp/` 看起来属于临时文件目录，但仍需先盘点。

处置原则：
- 不因目录名为 tmp 就直接删除；
- 先统计文件数量、大小和文件名；
- 再按单项或单目录授权处理。

### 4.6 带空格副本文件

带空格副本文件包括类似 `pyproject 2.toml`、`scripts/browser_button_smoke 2.py`、`tests/* 2.py` 等疑似重复副本。

处置原则：
- 需与正式路径文件做 hash / diff 比对；
- 不得未经备份删除；
- 对测试副本尤其要确认是否包含未合并的验证逻辑；
- 每次只处理一个文件或一组明确同类文件。

### 4.7 `tests/* 2.py`

`tests/* 2.py` 属于疑似测试副本，可能来自手工复制、系统复制或历史恢复过程。

处置原则：
- 先与对应正式测试文件做 hash / diff 比对；
- 若完全一致，可列入候选清理清单；
- 若不一致，需提取差异摘要并判断是否需要保留；
- 禁止未经备份删除测试副本。

### 4.8 `青天评标.app/`

`青天评标.app/` 属于应用包，高风险。

处置原则：
- 禁止直接删除；
- 禁止在未确认来源前移动；
- 先确认是否为可交付应用、历史构建产物或用户仍需使用的本地应用；
- 如需治理，应单独形成应用包处置方案。

## 6. 后续治理路线

建议按五阶段推进，避免一次性清理：

1. 第一阶段：只读清单化盘点。
   - 输出未跟踪项完整清单；
   - 统计各目录文件数量、目录数量、大小；
   - 标记高风险目录和疑似重复文件。
2. 第二阶段：按类别生成治理建议。
   - 区分缓存、运行态数据、业务数据、交付物、测试副本、应用包；
   - 每类给出保留、备份、比对、清理候选建议。
3. 第三阶段：对可疑副本做 hash / diff 比对。
   - 优先处理带空格副本文件；
   - 优先处理 `tests/* 2.py`；
   - 对不一致文件输出差异摘要，不直接删除。
4. 第四阶段：对必须保留内容做仓库外备份。
   - 备份目录应位于仓库外；
   - 记录备份路径和 hash；
   - 备份后仍需单独授权才可删除仓库内原文件。
5. 第五阶段：按单项、单目录、单指令处理。
   - 每次只处理一个明确目标；
   - 每次处理前后均输出 `git status --short`；
   - 不允许一次性清理。

## 7. 当前推荐下一步
